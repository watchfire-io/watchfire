# Changelog

## [6.0.0] Ember

### Added

- **Cross-project Insights rollup on the dashboard** — the dashboard home now ships a v6.0 Ember rollup card directly under the v4 Beacon status bar, answering the fleet-level productivity questions ("how many tasks across all my projects today, which agent did the most, total time / cost, success-rate trend") without forcing the user to open every project's InsightsTab in turn. The new `internal/daemon/insights/global.go` aggregator iterates every registered project (via `config.LoadProjectsIndex`), folds each completed task whose `CompletedAt` falls inside the requested window into one fleet-level set of counters — `tasks_total` / `tasks_succeeded` / `tasks_failed`, `tasks_by_day` (sorted chronologically, days with zero activity omitted), `top_projects` (sorted by completed-task count desc, ties broken by project name asc, capped at `MaxTopProjects = 5`, empty projects excluded so the rollup never carries dead pills), `agent_breakdown` (one row per backend, `success_rate = succeeded / count`, `avg_duration_ms` averaged from `CompletedAt - StartedAt` durations), `total_duration_ms`, `total_cost_usd`, and `tasks_missing_cost` — and persists the rollup to `~/.watchfire/insights-cache/_global.json` keyed by `(window_start, window_end)`. Cost is currently always 0 with every completed task counted as "missing cost" because per-task token / cost capture lives in task 0056 (Per-task metrics capture) and lands separately; the GUI partial-data caveat icon already reads that field, so when 0056 ships the rollup will surface real numbers without any wire change. Cache invalidation is a single-entry-point cascade: `InvalidateProjectCache(projectID)` cascades into `InvalidateGlobalCache()` and is called by `internal/daemon/server/server.go:processWatcherEvents` on every `EventTaskChanged` / `EventTaskCreated` / `EventTaskDeleted`, so any task-shape change anywhere in the fleet drops the rollup cache before the next dashboard refresh sees it. Proto extends `InsightsService` with `rpc GetGlobalInsights(GetGlobalInsightsRequest) returns (GlobalInsights)` plus the `DayBucket` / `AgentBreakdown` / `TopProject` / `GlobalInsights` messages (the `AgentBreakdown` shape is shared with the per-project rollup that lands in 0057, so 0057 can use it directly when it ships); the daemon handler in `internal/daemon/server/insights_service.go:GetGlobalInsights` delegates to the cached aggregator and `globalInsightsToProto` maps the Go struct to the wire shape one field at a time. The GUI rollup card (`gui/src/renderer/src/views/Dashboard/InsightsRollupCard.tsx`) is a sibling of the v4 Beacon `<StatusBar>` (placed directly underneath, not replacing it) and renders four pieces inside one rounded card: a four-cell KPI strip (Tasks · Success% · Time · Cost — the Cost cell carries an `<AlertTriangle>` warning icon with "N tasks missing cost" tooltip when `tasks_missing_cost > 0`), a 64-px-tall mini stacked-bar of tasks-per-day trimmed to the last 30 cells so a 90d window stays readable (succeeded green stacks atop failed red, peak-scaled so days are comparable not absolute), a top-projects pill list rendering each top project as a clickable button with its `project_color` dot — clicking a pill calls `useAppStore.selectProject(projectId)` to navigate to that project's view (where the per-project InsightsTab from 0057 will live) — and an agent stacked bar rendering `agent_breakdown` as proportional segments across one 8-px-tall horizontal bar with a colour-keyed legend underneath. The window selector chip group (7d / 30d / 90d / All) sits in the card header and persists the selection to `localStorage[wf-insights-window]` under the shared key `INSIGHTS_WINDOW_KEY` so the per-project InsightsTab from 0057 can read the same setting and the two views stay in sync without a server round-trip. The fetch hook `useGlobalInsights(window)` (`gui/src/renderer/src/hooks/useGlobalInsights.ts`) drops stale responses by sequence-number guard, so a slow 7d fetch can never overwrite a newer 30d response when the user flips windows quickly. Three render states cycle through `classifyRollup(insights, loading)`: `loading` shows a four-cell pulsing skeleton matching the KPI layout; `empty` ("No completed tasks in this window — run a task to populate insights.") covers fleet-wide zero state; `partial` is anything where `tasks_missing_cost > 0 || top_projects.length === 0` (renders normally with the cost caveat icon visible); `full` is the canonical complete render. Pure helpers — `formatDuration` ("71h 04m"), `formatCost` ("$23.71 (12 part)"), `formatPercent`, `dayBarHeights`, `agentSegmentWidths` (rounding residual absorbed in the last segment so widths always sum to 100%), `windowToRange` — live in `gui/src/renderer/src/lib/insights-rollup.ts` so the React component is purely a layout shell, mirroring the useExportReport pattern. Snapshot-style tests (`gui/src/renderer/src/lib/insights-rollup.test.mjs`) cover empty / partial / full classification + each KPI formatter + the day-bar peak-scaling + the agent-segment-rounding contract under twelve `node --test` assertions. The TUI gets a complementary fleet overlay (`internal/tui/insights.go`) reachable via the global `Ctrl+f` keybinding (matches the always-global pattern of `Ctrl+e` for export and `Ctrl+g` for global settings; the spec's `i` from a project list isn't applicable because the watchfire TUI is per-project — there's no project list view to anchor a context-scoped lowercase `i`): a rounded box with a `Fleet insights · last 30 days` header, a one-line KPI row, a unicode-block sparkline of tasks-per-day, a `Top proj` row listing each top project's name + count, an `Agents` row listing each backend's name + count, and a hint line `1 / 3 / 9 / 0 cycle window · Enter open project · Esc close`. Cycling between window presets re-issues `loadFleetInsightsCmd(conn, window)`; stale responses are dropped via a `fleetInsightsWindow` guard in the msghandler, matching the GUI's drop-stale pattern. Esc / q closes the overlay. Daemon-side aggregator unit tests (`internal/daemon/insights/global_test.go`) cover fleet totals, top-projects sort + tie-breaking + empty exclusion + 5-cap, window respect, agent breakdown success-rate, the cache round-trip per (window_start, window_end), and the cache cascade test asserting `InvalidateProjectCache(any)` deletes `_global.json`. Out of scope: per-day drill-down inside the rollup chart and cost forecasting / budget alerts (deferred to v6.x); export from the rollup card is already covered by the v6.0 task 0059 Export pill in the dashboard header.

- **Report export (CSV + Markdown)** — every Insights surface in v6.0 now offers one-click export through a shared `InsightsService.ExportReport` gRPC RPC (proto adds `ExportReportRequest` with a oneof scope of `project_id` / `global=true` / `single_task` ("`<project_id>:<task_number>`"), `ExportFormat` enum (`CSV = 0`, `MARKDOWN = 1`), and an optional `window_start` / `window_end` timestamp pair). Implementation lives in the new `internal/daemon/insights/` package: a render-from-data layer (`export.go`, `csv.go`, `markdown.go`) consumed by both production loaders (`LoadSingleTaskData`, `LoadProjectData`, `LoadGlobalData`, which walk the project's task YAMLs and the projects index) and golden-file tests. Markdown templates live in `internal/daemon/insights/templates/{single_task,project,global}.md.tmpl` so future tasks can tweak rendering without touching Go logic; CSV reports use `# section: <name>` header lines to delimit kpi / daily / agents / top_projects sub-tables in a single file. Filenames follow the canonical `watchfire-<scope>-<YYYY-MM-DD>.<ext>` convention (`watchfire-task-59-2026-05-02.md`, `watchfire-project-watchfire-2026-05-02.csv`, `watchfire-global-2026-05-02.md`); `MimeType()` returns `text/csv` / `text/markdown` so the browser-side `defaultDownload` helper can trigger a Blob URL download with the right hint. Six golden fixtures committed under `internal/daemon/insights/testdata/` (one per scope × format combination) lock down the byte-level output; an `-update` flag regenerates them after intentional template tweaks. The shared GUI `useExportReport()` hook in `gui/src/renderer/src/hooks/` exposes `exportSingleTask` / `exportProject` / `exportGlobal` callbacks plus a `loading` / `error` pair, and lands behind a single `<ExportPill>` component (`gui/src/renderer/src/components/ExportPill.tsx`) used by the Dashboard top-right and the ProjectView header — both surfaces drop a Markdown / CSV picker dropdown that calls the matching scope without each caller having to know about the gRPC client. The v6.0 DigestModal's Export button was rewired off its old clipboard-copy fallback onto the same hook, so clicking "Export Markdown" now produces a real `.md` file download covering the digest's window. The TUI gets a `Ctrl+e` global keybinding that opens an export-format picker overlay (`internal/tui/exportpicker.go` — Markdown highlighted by default, ↑/↓ to switch, Enter triggers, Esc cancels); the picker scopes to the current project and writes the rendered file to the project root with a `✓ Exported watchfire-project-foo-2026-05-02.md` status-bar confirmation that auto-clears after 3 s, with `OpenForSingleTask` and `OpenForGlobal` plumbing already in place for the per-task diff overlay (task 0055) and the cross-project Insights overlay (task 0058) to drop the same picker behind their `e` key once they land. PDF / HTML formats and scheduled exports beyond the weekly digest (task 0060) are deferred to v6.x.

- **Weekly digest notification (via Pulse bus)** — every Monday at 09:00 local (configurable to any weekday or `DAILY HH:MM` cadence) the daemon emits a `WEEKLY_DIGEST` notification through the v5.0 Pulse bus carrying a Markdown summary of the past 7 days across every registered project. The new `internal/daemon/server/digest.go:digestRunner` schedules with a re-armable `time.Timer` recomputed from `models.DigestSchedule.NextFire` each tick, so DST transitions don't drift — a "MON 09:00" schedule lands at the local wall-clock 09:00 even on the spring-forward / fall-back weekend. Catch-up logic on daemon startup: if the scheduled fire happened in the past 24 hours but no digest file exists for that calendar date, the runner fires once on boot; missed fires older than 24h are dropped to avoid spam after a long laptop sleep. The rendered Markdown is persisted to `~/.watchfire/digests/<YYYY-MM-DD>.md` (`config.GlobalDigestsDir()`) regardless of whether the gate suppresses the OS toast, so a user with `events.weekly_digest: false` (the default — opt-in per the v5.0 prefs UI default) can still pop open a digest from the in-app notification center after the fact. Gating reuses `models.ShouldNotify(NotificationWeeklyDigest, cfg, false, now)` extended with a new `defaults.notifications.events.weekly_digest` toggle (default `false`); master toggle, quiet hours, and project mute (N/A for fleet-wide digests) all compose through the same helper as TASK_FAILED / RUN_COMPLETE. Proto extends `enum NotificationKind` with `WEEKLY_DIGEST = 3` and `enum FocusTarget` with `FOCUS_TARGET_DIGEST = 3` plus a new `digest_date` field on `FocusEvent` so the tray and click handlers can route by ISO date. The notification body is a 280-char preview of the rendered Markdown plus the persisted file path, so the GUI's `gui/src/renderer/src/components/digest/DigestModal.tsx` (a new modal slotted into `App.tsx` and opened by clicking the OS toast, the tray's `📊 Weekly digest · last Mon` row, or a Digests entry in the bell-icon notification center) can render the full Markdown via a tiny in-tree renderer (no react-markdown dep needed for the constrained subset the template emits — headings, lists, bold, italic, code spans). The modal includes an Export pill (clipboard-copies the Markdown for now; will reuse the v6.0 task 0059 `useExportReport()` hook once that lands) and a "View in dashboard" button. The tray's `Notifications (N today) ▸` submenu now surfaces the latest digest as the topmost item with a "today / yesterday / last <Mon>" relative label; click sends a `FocusEvent{ Target: FOCUS_TARGET_DIGEST, DigestDate }` through the existing `DaemonService.SubscribeFocusEvents` stream which the GUI's `focus-store.ts` routes to the digest modal. Both the GUI's NotificationsSection (`gui/src/renderer/src/views/Settings/NotificationsSection.tsx`) and the TUI's globalsettings overlay (`internal/tui/globalsettings.go`) expose the master toggle (default off) plus a schedule selector (Mon 09:00 / Mon 18:00 / Fri 17:00 / Daily 09:00 presets, with the TUI cycling on Space and the GUI dropdown supporting custom values from a hand-edited YAML). Server-side validation in `settings_service.go:UpdateSettings` rejects malformed schedule strings via `models.ParseDigestSchedule` with the same error-toast pattern as quiet-hours validation. Tests cover DST spring-forward + fall-back boundaries (`TestNextFireDSTSpringForward`, `TestNextFireDSTFallBack` in `internal/daemon/server/digest_test.go`), catch-up inside / outside the 24h window (`TestRunnerCatchUpInsideWindow`, `TestRunnerCatchUpOutsideWindow`), gating with default-off and toggle-on (`TestRunnerGatingSuppressesEmit`, `TestRunnerGatingPassesWithToggleOn`), schedule-string parsing across valid weekday / DAILY / malformed inputs (`TestParseDigestSchedule`), and the new `Settings.Normalize()` defaulting for partial blocks (`TestNormalizeAddsDigestSchedule`). The settings round-trip test in `internal/config/settings_test.go` was updated to cover the new `weekly_digest` and `digest_schedule` fields end-to-end.

### Summary

The v6.0 Ember release composes three movements into a single arc — Inspect (task 0055 surfaced per-project history and metrics on the dashboard), Insight (tasks 0056–0058 added cross-project aggregation and the global-insights helper), and Iterate (tasks 0059–0060 close the loop with Markdown export templates and a proactive weekly digest notification). The user no longer has to remember to look at the dashboard: every Monday the daemon delivers a Markdown summary through the same Pulse channel that handles task-failure and run-complete toasts, with quiet-hours and master-toggle gating composing identically, and a click on the notification (or the new tray row, or a bell-icon notification-center entry) opens the persisted Markdown in an in-app modal with one-click export. v6.0 is the release where Watchfire stops being a tool the user opens to check on things and starts being one that opens itself, with curated summaries, when the things have changed in ways worth knowing about.

## [5.0.0] Pulse

### Fixed

- **In-app terminal couldn't find pnpm / volta / fnm-managed binaries (#32)** — opening a shell from the GUI's bottom panel produced a `pnpm: command not found` error even though `pnpm` ran fine in the user's native terminal, with the same trap repeating for any other tool installed under `~/.local/bin`, `~/.npm-global/bin`, Volta's shims, fnm's shims, or `~/.bun/bin`. Same root cause as the v2.0.0 Spark agent-auth fix: macOS / Linux GUI apps inherit a minimal `PATH=/usr/bin:/bin:/usr/sbin:/sbin` and don't read `~/.zshrc` / `~/.bashrc`, so user-installed tooling is invisible to anything spawned from the Electron main process. The v2.0.0 fix patched the daemon spawn path; the in-app terminal regressed to the broken state because `gui/src/main/pty-manager.ts:38` forked the user's shell with a raw `{ ...process.env }`. Both spawn paths now run through a single `gui/src/main/login-shell.ts` helper that runs `$SHELL -l -c env`, parses the output, plucks PATH plus the dev-tool roots a few CLIs reach for directly without consulting PATH first (`NVM_DIR`, `VOLTA_HOME`, `PNPM_HOME`, `BUN_INSTALL`, `DENO_INSTALL`, `CARGO_HOME`, `GOPATH`, `JAVA_HOME`, `LANG`, `LC_ALL`), times out at 3 s and falls back to `process.env` PATH-merged with a hardcoded list (`~/.local/bin`, `~/.npm-global/bin`, `/opt/homebrew/bin`, `/usr/local/bin`, `~/.cargo/bin`, `~/.bun/bin`) when the login-shell exec hangs / exits non-zero / produces no PATH, and caches the resolved env for the lifetime of the Electron process so per-PTY spawn cost stays at zero. `pty-manager.ts:createPty` is now async and awaits `loginShellEnv()` before forking; the `pty-create` IPC handler in `gui/src/main/ipc.ts` already returned a promise so no renderer-side change was needed. The daemon spawn path in `gui/src/main/daemon.ts` was refactored to call the same shared helper instead of its own duplicated copy, so a future bug-fix only needs to land in one place. Reporter also asked for a way to pick which shell binary the in-app terminal uses — a new `defaults.terminal_shell` global setting was added (proto field 8 on `DefaultsConfig`, persisted under `~/.watchfire/settings.yaml`'s `defaults.terminal_shell`, default empty for `$SHELL` autodetection) with a `Terminal shell` field in both the TUI global-settings overlay (`internal/tui/globalsettings.go`) and the GUI Defaults section (`gui/src/renderer/src/views/Settings/DefaultsSection.tsx`, with a `Browse…` button using `dialog.showOpenDialog` exposed through a new `browse-shell-binary` IPC handler). A non-empty value must point at an executable file; both the GUI (renderer-side `commitTerminalShell` round-trip via `SettingsService.UpdateSettings`) and the TUI (`globalsettings.go:validateExecutablePath`) reject malformed paths client-side for fast UX feedback, and the daemon repeats the X_OK check (`server/settings_service.go:validateExecutablePath`) as the source-of-truth gate before persisting. The shell-resolution order in `pty-manager.ts` is now: configured `defaults.terminal_shell` (re-read per PTY spawn so a settings change takes effect on the next new tab without an app relaunch — only used if the path still passes its X_OK check at spawn time, otherwise we silently fall back) → `process.env.SHELL` → `/bin/zsh`. Regression test in `gui/src/main/login-shell.test.mjs` (mirrors the `version.test.mjs` style — runs via `node --test`, no TS / Electron toolchain needed) covers env-output parsing including pathological lines (no `=`, leading `=`, embedded `=` in value), the relevant-env allowlist, the fallback-path merger asserting each fallback appears exactly once even when the user's PATH already contained some, the happy-path login-shell exec, the timeout fallback, the non-zero-exit fallback, and the no-PATH-output fallback. Daemon-side spawn semantics are unchanged from v2.0.0 — only the in-app terminal PTY path was touched. Fixes #32



- **Task-failure desktop notification (foundational v5.0 wiring)** — the daemon now fires a native OS notification the moment a task transitions to `status === 'done' && success === false`, telling the user *when they aren't looking at the window* — the dashboard's Beacon "needs attention" treatment (task 0041) only fires *if* they look. Emit is task-state-driven, not agent-state-driven: a manual edit of `0042.yaml` flipping `success: false` produces the same notification as an agent self-marking failed, because the hook lives in `internal/daemon/server/server.go:handleTaskChanged` (called from `processWatcherEvents` for both `EventTaskChanged` and `EventTaskCreated`), with the gating + emit logic factored into `internal/daemon/server/task_failed.go:emitTaskFailed`. Four standard preference gates short-circuit the emit via `models.ShouldNotify(NotificationTaskFailed, …)`: master `defaults.notifications.enabled`, per-event `defaults.notifications.events.task_failed`, per-project `notifications.muted`, and `defaults.notifications.quiet_hours`. The emit happens BEFORE the existing `StopAgentForTask` call so a notification is never lost to a slow agent shutdown. Title is `<project name> — task #NNNN failed`; body is the task title, optionally suffixed with `— <failure_reason>` when the YAML carries one, so the OS toast tells the user which task failed and why without making them open the GUI. Foundational infrastructure introduced for the whole Pulse release lives in a new `internal/daemon/notify/` package: a typed `Bus` with channel fan-out that mirrors the watcher / focus buses' slow-consumer-drop semantics (a stalled GUI subscriber never wedges the emitter), a `Notification` struct with stable `Kind` strings (`TASK_FAILED`, `RUN_COMPLETE`) that match the proto enum names so the JSONL fallback file is portable across surfaces, a `MakeID` helper that derives a stable `sha256(kind|project_id|task_number|emitted_at_unix)[:8]` so a duplicate burst (the watcher firing twice for one atomic write, or a manual save inside one second) collapses deterministically without coordinating with the emitter, and an `AppendLogLine` writer that produces `~/.watchfire/logs/<project_id>/notifications.log` — the headless-fallback file the tray's `Notifications (N today) ▸` submenu reads (task 0052) and any future post-hoc viewer can consume. The append happens unconditionally on every emit so a notification fired with no GUI client subscribed (CLI-only or tray-only sessions) still survives. Proto extends with `enum NotificationKind { TASK_FAILED, RUN_COMPLETE, STUCK_AGENT }` (`STUCK_AGENT` is reserved for a future task; only `TASK_FAILED` and `RUN_COMPLETE` ship in v5.0), `message Notification { id, project_id, task_number, title, body, emitted_at, kind }` mirroring the JSONL shape, and `service NotificationService { rpc Subscribe(...) returns (stream Notification) }` — both Go and TS bindings regenerated. The server registers a `notificationService{ bus }` at startup; the live gRPC stream and the JSONL writer share the same `*notify.Bus` instance constructed once in `server.New`. GUI side: the existing `gui/src/renderer/src/stores/notifications-store.ts` gains a `start()` / `stop()` lifecycle that opens the gRPC stream on connect and reconnects with a 2 s back-off on transient errors (mirroring the focus-store's reconnect pattern), feeds each incoming notification into the existing 50-entry ring buffer + sound-playback pipeline, and hands the metadata off to a new `notifications:emit` IPC handler in `gui/src/main/ipc.ts` that calls `new Notification({ title, body, silent: true }).show()` — `silent: true` because the renderer is the source of sound truth (task 0051's bundled WAVs play in the renderer when the user has enabled them), so the OS and the renderer never double-fire. Click → focus the window via `BrowserWindow.show()/focus()` and emit a `notifications:click` IPC event back to the renderer, which `App.tsx` turns into a `useAppStore.requestFocus({ projectId, target: 'tasks', taskNumber })` so the GUI lands on the failing project's TasksTab. Tests cover bus fan-out + slow-consumer non-blocking emit + nil-bus-safe Subscribe/Emit, `MakeID` determinism + per-second collapse, JSONL append + multi-line append + missing-project-id rejection (`internal/daemon/notify/bus_test.go`), the four short-circuit gates on `emitTaskFailed` plus the non-done / success=true skip paths (`internal/daemon/server/task_failed_test.go`), and an integration test that writes a `status: done, success: false` YAML to a registered project, drives `handleTaskChanged` directly, and asserts the notification arrives on a real `*notify.Bus` subscriber the same way `NotificationService.Subscribe` consumes it (`internal/daemon/server/notification_integration_test.go`).

- **Run-complete desktop notification when an autonomous run drains** — the daemon now fires exactly one `RUN_COMPLETE` notification at the falling edge of every autonomous run on a project (single-task `task` mode, or any chained `start-all` / `wildfire` run that finishes draining its queue). Title is `<project name> — run complete`; body is `N tasks done · M failed` where the counts are derived from tasks whose `status === 'done'` and `updated_at` falls within the run window. The "run window" is bounded by the rising edge of `agentStatus.isRunning` for the project (a new `RunningAgent.RunStartedAt time.Time` field stamped in `internal/daemon/agent/manager.go:StartAgent`, preserved across chain restarts via `StartOptions.RunStartedAt` so the second-and-later agents in a `start-all` / `wildfire` chain inherit the original anchor instead of re-stamping their own) and the falling edge of the `monitorProcess` exit path (single-task non-chain cleanup, chain-out-of-tasks "no more tasks" branch, and chain-abort branches like `userStopped` / `hasIssue` / restart-limit-reached, the last of which deliberately emits `RUN_COMPLETE` for the autonomous portion *before* transitioning into chat mode so the user gets a clear "the autonomous part finished, the chat is a hand-off" signal). Two filters short-circuit the emit on top of the four existing notification gates: chat / generate-definition / generate-tasks modes are excluded outright (`runCompleteApplicable`) — these never autonomously chain through tasks, so their falling edge isn't a "run" in the v5.0 Pulse sense — and a window with zero done + zero failed task transitions is dropped silently (`runCompleteBody` returns the empty string), so a user who manually stopped an agent before it did anything doesn't get a confusing "0 tasks done · 0 failed" toast. Both filter functions live in `internal/daemon/agent/run_complete.go` as pure helpers and are exercised directly by `internal/daemon/agent/run_complete_test.go` (mixed-window happy path, chat-mode skip, three flavours of empty-window skip). The four standard preference gates from task 0049 are layered on top via `models.ShouldNotify(NotificationRunComplete, …)`: master `defaults.notifications.enabled`, the new per-event `defaults.notifications.events.run_complete` toggle (default true; UI lands in 0053), per-project `notifications.muted`, and `defaults.notifications.quiet_hours`. Relationship to `TASK_FAILED` (task 0049): a single-task run that fails emits BOTH a `TASK_FAILED` (one task ended badly) AND a `RUN_COMPLETE` with body `0 tasks done · 1 failed` (the run also ended) — they're two different signals on independent toggles, deliberately, so a user who only cares about run-end can mute task-failed and still get one toast per drained run. A new `internal/daemon/notify/` package introduces a typed `Bus` with channel fan-out (slow-consumer drop semantics matching the watcher / focus buses), a `Notification` struct mirroring the JSONL schema the tray's `LoadRecentNotifications` already reads, a stable `MakeID` derived from `sha256(kind|project_id|task_number|emitted_at_unix)` so a duplicate burst dedupes deterministically, and an `AppendLogLine` writer that produces `~/.watchfire/logs/<project_id>/notifications.log` — the same headless-fallback file the tray's "Notifications (N today) ▸" submenu reads (task 0052), so a `RUN_COMPLETE` notification fired with no GUI client subscribed is still reachable from the tray with the click-target distinct from `TASK_FAILED`: the tray menu builder (`internal/daemon/tray/menu.go`) already routes `RUN_COMPLETE` rows to `focus.TargetMain` (project's main / dashboard view) and `TASK_FAILED` rows to `focus.TargetTasks` (project's TasksTab), so a daemon click-through lands the GUI on the correct view per the spec without any further routing work. The bus is wired into the daemon at server startup (`server.New` calls `notify.NewBus()` and passes it to `agentMgr.SetNotifyBus`); 0049's `NotificationService.Subscribe` RPC will consume the same instance.

- **Bundled notification sounds with per-event + master toggles** — task-completion and task-failure notifications now ship with two short, embedded WAV cues so the user actually hears that something finished without staring at the tray. `assets/sounds/task-done.wav` (~500 ms, ~22 KB) is a neutral / positive ascending two-note ding played on `RUN_COMPLETE`; `assets/sounds/task-failed.wav` (~580 ms, ~25 KB) is a subdued descending two-note tone played on `TASK_FAILED`, intentionally calm rather than alarming. Both files are mono, 22050 Hz, 16-bit PCM, well under the 50 KB / 700 ms ceilings in the v5.0 Pulse spec, and were generated from scratch by the project (pure sine tones with simple ADSR envelopes — no third-party samples; see `assets/sounds/README.md` for the source / licence). The renderer ships build-time mirrors of both files in `gui/src/renderer/public/sounds/` (canonical files at the repo root, public-dir copies for the bundle) and references them via absolute URLs at runtime. The new `gui/src/renderer/src/stores/notifications-store.ts` (the gRPC stream subscriber lands in task 0049) preloads both `Audio` instances at store creation — touched eagerly via `useNotificationsStore.getState()` at module-top in `App.tsx` — so the first playback after a notification fires lands without disk-fetch latency. The pure playback decision lives in a separate file `gui/src/renderer/src/stores/notifications-sound.ts:shouldPlaySound(kind, prefs)` so it can be unit-tested without Vite, and `playSoundForKind(kind, prefs, audios)` does the play() with `audio.currentTime = 0` (so a rapid second notification restarts the sound cleanly) and a swallowed `play()` rejection (autoplay policy compliance — Chrome/Electron block autoplay until first user interaction; in practice the user has interacted long before a notification fires, but the catch keeps the console clean). Four prefs are honoured under `defaults.notifications.sounds` in `~/.watchfire/settings.yaml`: `enabled` (master toggle), `task_failed` and `run_complete` (per-event toggles), and `volume` (0.0–1.0 float, applied via `audio.volume = ...` before play and clamped if a hand-edited YAML pushes it out of range). Electron's `before-quit` handler in `gui/src/main/index.ts` already calls `destroyAllPtys()` from `pty-manager.ts`, so the global cleanup path on app quit is unchanged. Architecturally this flips the bottom panel from "scoped to the active project view" to "global session pool with per-project filtering," which is the only model where long-lived dev servers and tail loops can coexist with project hopping. `ARCHITECTURE.md:907` is updated to match the new contract

### Changed

- **Dashboard auto-sorts projects by activity** — the project grid (and dense list layout) on the Dashboard no longer renders projects in raw `~/.watchfire/projects.yaml` `position` order. With many projects in flight at once, that order buried the projects that actually needed eyes — a freshly-failed task on project #12 looked identical at a glance to a long-idle project pinned at the top. `gui/src/renderer/src/views/Dashboard/Dashboard.tsx` now derives a `sortedProjects` array via `sortProjectsByActivity` (in the new `gui/src/renderer/src/lib/dashboard-filters.ts` predicate module that task 0037's filter chips will share), bucketing projects into four activity groups and floating the most relevant to the top: **(0) needs attention** — any non-deleted task with `status === 'done' && success === false`; **(1) working** — `agentStatuses[projectId].isRunning` is true; **(2) has ready tasks** — at least one non-deleted `status === 'ready'` task and not in groups 0 or 1; **(3) idle** — everything else. Within each group the input array's index is used as the final tiebreaker so positions don't jitter on every render — combined with V8's stable `Array.prototype.sort`, the ordering is doubly stable. The pipeline is structured `filter → sort` so when task 0037's chip selection lands it slots in cleanly: filter the projects list first, then pass the result through `sortProjectsByActivity`. Both the grid (`rectSortingStrategy`) and the list layout (`verticalListSortingStrategy`) drive their `SortableContext` items off `sortedProjects`, so the dnd-kit drag-handle visuals match what the user sees. When the activity-sorted order differs from the underlying `position` order, a small `Sorted by activity` muted italic label renders at `text-[11px]` between the layout toggle and the grid, with a `Needs attention → working → ready → idle` tooltip — when nothing is firing and every project is idle, the underlying order wins, so the hint disappears entirely. There is intentionally no toggle to disable the sort for now (a manual override stays out of scope until the multi-project monitoring story actually demands it). The predicate helpers (`hasFailedTask`, `hasReadyTask`, `isProjectWorking`, `isProjectIdle`, `projectActivityGroup`, `sortProjectsByActivity`, `projectOrderDiffers`) live in `dashboard-filters.ts` precisely so task 0036's aggregate status bar and task 0037's filter chips compute their counts from the same definitions — drift between "needs attention" on a card border, in the status bar, in a chip count, and in the auto-sort grouping is now structurally impossible

### Fixed

- **`watchfire update` failed across filesystems on Linux (#25)** — a Fedora user reported `failed to update CLI: install new binary: rename /tmp/watchfire-update-2058306240 /home/madoke/.local/bin/watchfire: invalid cross-device link` when upgrading from 0.9.0 to 1.0.0. Root cause: `updater.DownloadAsset` staged the downloaded binary under `os.TempDir()` (on Fedora/Ubuntu `/tmp` is a separate `tmpfs` filesystem) and `updater.ReplaceBinary` finished the install with `os.Rename(newPath, destPath)`, which on Linux boils down to the `renameat2(2)` syscall and returns `EXDEV` across filesystems. Fix: `DownloadAsset` now takes a `preferDir` argument and stages the download inside the install directory itself (e.g. `~/.local/bin/.watchfire-update-XXXXXX`) so the final rename is always same-filesystem and therefore atomic; the CLI `update` command resolves the install dirs for both the CLI (`os.Executable()` + `EvalSymlinks`) and the daemon (`findDaemonBinary` + `EvalSymlinks`) up front and passes each one as the preferred staging dir. As belt-and-suspenders, `ReplaceBinary` keeps its atomic-rename semantics even if a future caller ever stages the binary elsewhere: it detects the cross-dir case, copies+fsyncs the staged binary into the install dir, removes the original, and finishes with a same-dir rename — the final swap is still a single atomic operation so a concurrent `watchfire` invocation never observes a half-written binary. Exec perms (`0o755`) are applied before the rename so the final binary lands already executable. Regression test in `internal/updater/updater_test.go` covers the same-dir fast path, the cross-dir EXDEV fallback (no leftover staging file, original temp cleaned up), exec-perm preservation, and the `preferDir` fallback to `os.TempDir()` when the install dir is unusable. The daemon binary update path goes through the same `DownloadAsset`/`ReplaceBinary` pair so it inherits the fix automatically. Fixes #25
- **Task list rotated in projects with many tasks (#28)** — a Fedora user with 47 tasks reported the task list starting at `0017`, running to `0047`, then wrapping back to `0001`. Root cause: the task manager's `ListTasks` sorted primarily by the legacy `position` field and only used `task_number` as a tiebreaker, and the TUI re-sorted within each status group with the same key. When a project had tasks split across status groups (16 done + 31 ready in the reporter's case), the section headers — `Ready` renders before `Done` in the TUI, and `In Development` before `Done` in the GUI — placed `0017…0047` above `0001…0016`, producing the "rotation" despite position and task_number matching. Sorting is now canonical across every surface: the task manager returns tasks strictly descending by `task_number` (newest first), the CLI `task list`, TUI task list, and GUI TasksTab all rely on that order without re-sorting, and the legacy `position` field is ignored at read time. Regression test in `internal/daemon/task/manager_test.go` seeds 25 mixed-status tasks with shuffled `position` values and asserts the returned order is strictly descending by `task_number`. Fixes #28
- **GUI prompted to update CLI on every launch (#30)** — the `cli-installer` version check compared the installed CLI's version string against `app.getVersion()` with strict `!==`, which tripped on trailing whitespace, pre-release suffixes, and ANSI hyperlinks that escaped the old CSI-only stripper; on Linux it also read the wrong binary because the search-dir order (`/usr/local/bin` → `~/.local/bin`) put the system path ahead of the user path that `installCLI()` actually writes to, so a stale 2.0.0 binary in `/usr/local/bin` kept re-triggering the prompt even after the user clicked Install. Version parsing now lives in a pure `gui/src/main/version.ts` module with a broader ANSI stripper (CSI + OSC + other ESC), a proper semver-aware comparator (trims whitespace, drops leading `v`, ignores build metadata), and the Linux/macOS search order now matches the install target with a PATH fallback (`command -v`) for rpm/deb/Linuxbrew installs. Regression tests in `gui/src/main/version.test.mjs` cover the representative Linux and macOS outputs (run with `node --test`)
- **Newly-installed agents invisible in GUI/TUI pickers (#29)** — a Fedora user installed Codex while Watchfire was running and neither the TUI nor the GUI surfaced it in the agent picker until `project.yaml` was hand-edited. The architectural root cause was that agent discovery had been implicitly coupled to binary availability — a backend whose `ResolveExecutable` returned an error could get hidden entirely, so a freshly-installed CLI stayed invisible until the resolver happened to find it and the UI re-enumerated. The registry is now the sole source of truth for pickers: `SettingsService.ListAgents` always returns every registered backend (Claude Code, Codex, Gemini, opencode, Copilot) and exposes a new `AgentInfo.available` boolean as a display-time hint only. GUI (`SettingsTab`, `AgentPathsSection`, `DefaultsSection`, `StepAgent`) and TUI (`settings.go`, `taskform.go`, `globalsettings.go`) pickers now append a `(not installed)` suffix to unavailable agents instead of omitting them, so users can select a backend they're mid-installing and get a clear error at spawn time rather than a silent absence. Linux fallback paths for Claude, Codex, Gemini, opencode, and Copilot were simultaneously broadened to cover `/usr/bin/<name>` (distro packages like Fedora `dnf`) and `~/.npm-global/bin/<name>` where they were missing. Regression test in `internal/daemon/server/settings_service_test.go` registers a fake backend whose binary always fails to resolve and asserts it still appears in `ListAgents` with `Available=false`. Fixes #29

## [0.1.0] Ember

Watchfire orchestrates coding agent sessions (starting with Claude Code) based on project specs and tasks. Define what you want built, break it into tasks (or have agents do it), and let agents work through them autonomously — with full visibility into what's happening. Or just turn on wildfire mdoe and let you agents do it all for you.

### Daemon (`watchfired`)

The always-on backend that manages everything:

- **Agent orchestration** — Spawns coding agents in sandboxed PTYs with terminal emulation, one task per project, multiple projects in parallel
- **Git worktree isolation** — Each task runs in its own worktree (`watchfire/<task_number>`), auto-merged back on completion with conflict detection
- **macOS sandbox** — Agents run inside `sandbox-exec` with restricted filesystem/network access
- **File watching** — Real-time detection of task completion and phase signals via fsnotify, with polling fallback for reliability
- **Session logs** — Agent sessions logged to `~/.watchfire/logs/` with JSONL transcripts from Claude Code (clean conversation format) and PTY scrollback fallback
- **System tray** — Menu bar icon showing daemon status, active agents with colored project dots, and quick stop/quit actions
- **Secrets folder** — `.watchfire/secrets/instructions.md` for providing agents with external service credentials and setup instructions, injected into the system prompt
- **Issue detection** — Monitors agent output for auth errors (401, expired tokens) and rate limits (429), with real-time notifications to clients
- **gRPC + gRPC-Web** — Single port serves both native gRPC (CLI/TUI) and gRPC-Web (Electron GUI)
- **Auto-discovery** — Writes connection info to `~/.watchfire/daemon.yaml` so clients find it automatically

### CLI (`watchfire`)

Project-scoped command-line interface:

- `watchfire init` — Initialize a project (git setup, `.watchfire/` structure, `.gitignore`, interactive config)
- `watchfire task add|list|edit|delete|restore` — Full task CRUD with soft delete/restore
- `watchfire definition` — Edit project definition in `$EDITOR`
- `watchfire settings` — Configure project settings interactively
- `watchfire agent start [task|all]` — Start agent in chat, single-task, or run-all-ready mode
- `watchfire agent wildfire` — Autonomous three-phase loop: execute ready tasks → refine drafts → generate new tasks → repeat
- `watchfire agent generate definition|tasks` — One-shot generation commands
- `watchfire daemon start|status|stop` — Daemon lifecycle management
- `watchfire update` — Self-update from GitHub Releases
- **Terminal attach** — Raw PTY streaming with resize handling and Ctrl+C forwarding
- **Self-healing project index** — Auto-registers projects, updates moved paths, reactivates archived projects

### TUI (`watchfire` with no args)

Interactive split-view terminal interface:

- **Split layout** — Task list (left) + agent terminal (right) with draggable divider
- **Left panel tabs** — Tasks (grouped by status), Definition (read-only + `$EDITOR`), Settings (inline form)
- **Right panel tabs** — Chat (live agent terminal), Logs (session history viewer)
- **Agent modes** — Chat, task, start-all, and wildfire with phase display (Execute/Refine/Generate)
- **Issue banners** — Auth required and rate limit detection with recovery guidance
- **Keyboard navigation** — Vim-style (`j/k`), arrows, tab switching (`1/2/3`), panel focus (`Tab`)
- **Mouse support** — Click to focus/select, scroll, drag divider to resize
- **Task management** — Add, edit, status transitions (draft/ready/done), soft delete — all from the keyboard
- **Auto-reconnect** — Reconnects to daemon on disconnect with status indicator
- **Help overlay** — `Ctrl+h` for full keybinding reference

### GUI (Electron)

Multi-project desktop application:

- **Dashboard** — Project cards with task counts, status dots, active task display
- **Project view** — Tasks, Definition, Secrets, Trash, Settings tabs with collapsible right panel (Chat, Branches, Logs)
- **Add Project wizard** — Three-step flow: project info → git config → definition
- **Branch management** — View, merge, delete, and bulk-manage worktree branches
- **Remove project** — Unregister projects from sidebar context menu, dashboard card, or settings tab (stops agents, no files deleted)
- **Agent terminal** — Live streaming via gRPC-Web with input support
- **Global settings** — Defaults, appearance (system/light/dark theme), agent path config, update preferences
- **Daemon lifecycle** — Auto-restarts daemon if it dies, handles binary updates gracefully

### Agent Modes

| Mode | Description |
|------|-------------|
| **Chat** | Free-form conversation with the agent at project root |
| **Task** | Work on a specific task in an isolated worktree |
| **Start All** | Run all ready tasks in sequence, one at a time |
| **Wildfire** | Fully autonomous loop: execute → refine → generate → repeat until done |
| **Generate Definition** | One-shot: agent analyzes codebase and writes project definition |
| **Generate Tasks** | One-shot: agent reads definition and creates task files |

### Task Lifecycle

```
draft → ready → done (success ✓ or failure ✗)
```

- Tasks are YAML files in `.watchfire/tasks/`
- Agents detect completion by writing `status: done` to the task file
- Daemon auto-merges the worktree branch, cleans up, and chains to the next task
- Merge conflicts abort the chain to prevent cascading failures

### Build & Distribution

- **macOS DMG** — Universal binary (arm64 + amd64) with GUI, CLI, and daemon bundled
- **Code signing & notarization** — Developer ID certificate with hardened runtime
- **Homebrew** — `brew tap watchfire/tap && brew install watchfire`
- **Auto-update** — GUI via `electron-updater`, CLI via `watchfire update`, daemon checks on startup
- **CI/CD** — GitHub Actions: lint, test, build matrix (arm64/amd64), sign, notarize, draft release

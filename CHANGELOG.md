# Changelog

## [8.0.0] Echo

### Summary

v8.0 "Echo" closes the half-loop v7.0 Relay left open: with v7.0 the daemon could *speak* to outside services (TASK_FAILED to Slack, RUN_COMPLETE to Discord, generic webhooks, GitHub auto-PR) but couldn't *hear* a reply. v8.0 lands an inbound HTTP server (`internal/daemon/echo/`) bound to a user-configurable address (default `127.0.0.1:8765`, optional public reach via a tunneled URL) with reliable-delivery semantics, an LRU+TTL idempotency cache (1000 entries / 24h), provider-specific signature verification (GitHub HMAC-SHA256, Slack v0 HMAC + 5-minute timestamp drift, Discord Ed25519 — all constant-time), and a 1-MiB per-request body cap. The Discord interactions endpoint is the first concrete handler on top of the framework; the shared transport-agnostic `commands.Route(...)` returns a `CommandResponse{text, blocks, ephemeral, in_channel}` so the same router serves both Slack and Discord identically. Three movements: **Listen** (HTTP server + signature verification + idempotency), **React** (concrete provider handlers + `task.{Retry, Cancel, MarkDoneIfNotAlready}` lifecycle helpers + `commands.Route` router), **Trust** (per-provider signature verifiers running before any state mutation, secrets stored in the OS keyring with a graceful file-store fallback). The v7.0 outbound flow is unchanged; `models.InboundConfig` is purely additive — empty config = no inbound, no behaviour change.

### Added

- **Inbound HTTP server framework + signature verification + idempotency cache** — `internal/daemon/echo/server.go` exposes `New(InboundConfig, *log.Logger) *Server` returning an `http.ServeMux`-backed server that binds the configured listen address (default `127.0.0.1:8765`), serves until ctx cancels, drains in-flight handlers for 5 seconds on graceful shutdown, and registers per-provider handlers via `RegisterProvider(method, path, handler)`. The `wrap(...)` middleware applies a 1 MiB `http.MaxBytesReader` cap + panic recovery to every handler so an oversized body returns 413 before any memory is allocated and a handler panic returns 500 instead of crashing the daemon. The unauthenticated `GET /echo/health` endpoint returns 200 with `{"listening": true, "version": "8.0.0"}` so upstream tunnels (ngrok / Cloudflare Tunnel) can confirm the daemon is up without exposing any secrets. Per-provider signature verifiers in `internal/daemon/echo/verify.go`: `VerifyGitHub(secret, body, header)` runs HMAC-SHA256 against the `sha256=<hex>` header value through `hmac.Equal` (constant-time); `VerifySlack(secret, timestamp, body, sig)` runs HMAC-SHA256 over `v0:<timestamp>:<body>` with a 5-minute `|now - timestamp|` drift window; `VerifyDiscord(publicKey, timestamp, body, sig)` runs `ed25519.Verify(publicKey, timestamp || body, sig)`. Each verifier rejects malformed-prefix, malformed-hex, empty-secret, and out-of-window cases with a generic 401 so a probe loop cannot use the response to differentiate failure modes. The idempotency cache (`internal/daemon/echo/idempotency.go`) is a thread-safe LRU+TTL store keyed by the provider-specific delivery id (`X-GitHub-Delivery` for GitHub, `client_msg_id` / `trigger_id` for Slack, interaction `id` for Discord) — `Seen(key)` returns true if the key was inserted within the past TTL, false otherwise (and inserts on miss); default 1000 entries / 24h TTL; LRU eviction on overflow. Per-process state — not persisted across restarts. The new `models.InboundConfig` shape (`internal/models/integrations.go`) holds `ListenAddr`, `PublicURL`, `Disabled`, four `*Ref` keyring references (`GitHubSecretRef`, `SlackSecretRef`, `DiscordPublicKeyRef`, `DiscordBotTokenRef`), and `DiscordAppID` — purely additive to `IntegrationsConfig` so v7.0 installs upgrade with no behaviour change. Tests cover signature verification across every provider + every rejection path (`verify_test.go`), idempotency under fresh / duplicate / expired / eviction / refresh-on-hit / empty-key paths (`idempotency_test.go`), and `RecordDelivery` / `LastDelivery` per-provider tracking (`server_status_test.go`).

- **Per-task lifecycle helpers + transport-agnostic command router** — `internal/daemon/echo/commands.go` ships the `Route(ctx, cmd, subcmd, rest, CommandContext) CommandResponse` shared command router that powers every inbound slash-command transport. Three commands: `status` (lists in-flight tasks across configured projects, scoped per `CommandContext.GuildID` / `TeamID` so a daemon serving multiple guilds doesn't leak projects across them), `retry <task>` (re-queues a previously-failed task by flipping its YAML `status: ready` + clearing `success` + bumping `agent_sessions`), and `cancel <task>` (stops a running agent if `in_progress`, else marks `done` with `success: false, failure_reason: "cancelled via inbound"`). `CommandResponse` carries `text`, `blocks` (Block Kit-compatible — header / section / context / divider), `ephemeral` (private response, maps to Slack ephemeral / Discord `flags: 64`), and `in_channel` (publicly-visible response). The router takes its task / project lookups through injected callbacks (`FindProjects`, `LookupTask`, `ListTopActiveTasks`, `Retry`, `Cancel`) so it stays transport-agnostic — Discord and Slack handlers each pass their own `CommandContext` and render the response in their native format. The status command returns one `(project header → tasks → divider)` section per mapped project, capped at three active tasks per project. The retry / cancel commands return an in-channel confirmation on success (`✅ Retrying task #0042 — build the thing`) and an ephemeral error on failure (missing task, lifecycle error) so private failures stay private and public successes stay public.

- **Discord interactions endpoint (slash commands mirror) + framework backfill** — v8.0 Echo's third inbound transport lands as `internal/daemon/echo/handler_discord.go`, exposing `POST /echo/discord/interactions` so a Discord bot can send `/watchfire status [project]`, `/watchfire retry <task>`, and `/watchfire cancel <task>` slash commands from any guild it is invited to and have them route into the daemon's task lifecycle helpers without leaving the channel. The handler enforces Discord's signature contract end-to-end: every request must carry the `X-Signature-Ed25519` and `X-Signature-Timestamp` headers, and `internal/daemon/echo/verify.go::VerifyDiscord` runs `ed25519.Verify(publicKey, timestamp || body, sig)` before any router dispatch — failing the verification disables the app at registration time, so getting this right is the foundational step. The verifier is layered behind a 5-minute replay window (`echo.ReplayWindow`) that rejects requests whose timestamp drifts past Discord's typical re-delivery horizon, and a process-level LRU+TTL idempotency cache (`echo.NewCache`, default 1000 entries / 24h TTL keyed by Discord's per-request `id`) ensures a re-delivery — Discord retries an interaction up to a handful of times if the initial response is delayed — produces a single ack rather than a double-action. The handler distinguishes four interaction types: `type: 1` (PING) → returns `{type: 1}` PONG, the only response shape Discord accepts at endpoint-verification time; `type: 2` (APPLICATION_COMMAND) → flattens `data.options[]` into a single `rest` string, dispatches to the shared `echo.Route(ctx, cmd, subcmd, rest, CommandContext)` router (which lands `CommandResponse{text, blocks, ephemeral, in_channel}`), and renders the result as Discord's `type: 4` CHANNEL_MESSAGE_WITH_SOURCE JSON via `internal/daemon/echo/discord_render.go::RenderInteraction`; `type: 3` MESSAGE_COMPONENT, `type: 4` AUTOCOMPLETE, and `type: 5` MODAL_SUBMIT → polite ephemeral "Not supported in v8.0" reply rather than a hard rejection so the bot stays alive in the user's server even if they wire up extra UI later. `RenderInteraction` translates the transport-agnostic Block Kit-compatible block types into Discord's embed schema: `header` blocks become embed `title`, successive `section` blocks concatenate into the embed `description` (joined with `\n\n`), `context` blocks become the embed `footer.text`, and `divider` blocks split the response into a fresh embed — so a status response covering three projects renders as three embeds, one per project, matching the visual rhythm of the v7.0 Relay outbound Discord adapter. `Ephemeral: true` maps to Discord's `data.flags = 64` EPHEMERAL flag; `InChannel: true` is the publicly-visible counterpart used by `retry` and `cancel` confirmations. Description rendering applies a defensive 4000-rune cap with an ellipsis suffix, mirroring the v7.0 outbound adapter's `discordEmbedDescriptionLimit` so a stray multi-page section never trips Discord's 4096-rune hard limit. The shared command router in `internal/daemon/echo/commands.go` is transport-agnostic: `Route(ctx, cmd, subcmd, rest, CommandContext)` takes injected callbacks (`FindProjects` / `LookupTask` / `ListTopActiveTasks` / `Retry` / `Cancel`) so the same router serves Slack and Discord identically; the `CommandContext.GuildID` / `TeamID` fields scope the project lookup per-request so a daemon serving multiple guilds does not leak projects across them. The status command renders a header block + one (project header → tasks → divider) section per mapped project, capped at three active tasks per project, plus a final `As of <RFC3339>` context footer. The retry / cancel commands return an in-channel confirmation on success ("✅ Retrying task #0042 — build the thing (alpha)") and an ephemeral error on failure (missing task, lifecycle error) so private failures stay private and public successes stay public. Project mapping extends `internal/models/integrations.go::DiscordEndpoint` with a `GuildID` field — inbound interactions route to projects whose Discord endpoints share the requesting `guild_id`; an empty mapping renders an ephemeral "no Watchfire projects mapped to this Discord guild" reply pointing the user at Settings. The new `internal/models/integrations.go::InboundConfig` shape is purely additive (empty / zero-value = no inbound listener, preserving v7.0 behaviour) and carries `ListenAddr` (default `127.0.0.1:8765`), `PublicURL` (user-supplied display only — paste an ngrok / Cloudflare-tunnel URL), `DiscordPublicKeyRef` (keyring reference for the Ed25519 application public key), `DiscordAppID` (for command registration), and `DiscordBotTokenRef` (for command registration). Slash-command registration is deliberately a manual step in v8.0 — the new `watchfire integrations register-discord <guild_id>` CLI POSTs the three command schemas (`status` with optional `project` filter, `retry` and `cancel` each requiring a `task` arg) to `https://discord.com/api/v10/applications/<app_id>/guilds/<guild_id>/commands` and is idempotent on re-run because Discord's commands API upserts on POST when a command name already exists. The CLI surfaces a per-command status glyph (`✓` / `✗ HTTP 4xx`) so a partial failure (e.g. missing `applications.commands` scope on one command) is visible without obscuring the rest. Tests: `internal/daemon/echo/verify_test.go` covers GitHub HMAC-SHA256, Slack v0, and Discord Ed25519 signature verification with valid + tampered + expired + malformed-prefix + malformed-hex + empty-secret paths, plus a non-numeric-timestamp Discord case asserting the signature check still passes when the timestamp doesn't parse as Unix-seconds (Discord's signing surface is byte-exact on the timestamp string); ### Migration

Empty `InboundConfig` = no behaviour change. v7.0-and-earlier installs upgrade to v8.0 with the master toggle effectively off (no provider secret has been entered). The Echo HTTP server still binds and serves the unauthenticated `/echo/health` endpoint (it returns `{"listening": true, "version": "8.0.0"}` and exposes no secrets), but every concrete provider handler is gated on a secret resolution that returns 503 if the secret is not configured. To enable inbound: enter the relevant provider's secret in **Settings → Inbound (Echo)** or via `~/.watchfire/integrations.yaml`'s new `inbound:` block, paste the matching `<public-url>/echo/<provider>` URL into the upstream provider's webhook config (Discord application interactions endpoint URL / GitHub repository webhook URL / Slack app slash-command request URL), and the loop closes the next time the provider fires. No daemon restart is required after entering a secret — `SaveInboundConfig` calls `restartEchoServer` which tears down + rebinds the listener in-process, and the GUI's status pill flips green on the next 5-second poll.

### Known issues

- **Concrete `handler_github.go` for `pull_request.closed` events did not ship in v8.0.** The framework, signature verifier, idempotency cache, and registration scaffolding all shipped, but the dedicated handler that translates GitHub's webhook body into a `task.MarkDoneIfNotAlready` call is filed as v8.x follow-up `echo8gith2`. Until that ships, the auto-PR loop closes manually: the user merges the PR on github.com and the local task stays at whatever status the agent left it (`done` + `success: true` is the typical case for a PR-opened-on-success flow).
- **Slack HTTP transport on top of the shared `commands.Route` did not ship in v8.0.** The shared transport-agnostic command router shipped (Discord exercises it end-to-end), but the per-channel HTTP handler that translates `CommandResponse` → Slack response payload is filed as v8.x follow-up `echo8sltr`. Until it ships, `/watchfire status` / `retry` / `cancel` work in Discord but not in Slack.

## [7.0.0] Relay

### Added

- **Outbound delivery framework + generic webhook adapter** — v7.0 Relay's foundational plumbing now lives in `internal/daemon/relay/` so every per-channel adapter (Slack / Discord / generic webhook) iterates the same `Adapter` interface (`ID()` / `Kind()` / `Supports(notify.Kind)` / `Send(ctx, Payload) error`) declared in `relay.go`, behind a single `Dispatcher` that owns the live fan-out from the v5.0 Pulse `notify.Bus` to every configured endpoint. The dispatcher (`internal/daemon/relay/dispatcher.go`) subscribes once at construction (so any Emit between NewDispatcher and Run lands in the buffered subscription channel rather than racing the registration), runs a `Run(ctx)` goroutine that pulls each Notification, resolves it through a pluggable `PayloadResolver` (the daemon's resolver in `internal/daemon/server/relay_dispatch.go::resolveNotificationPayload` looks up project name + color via the projects index and task title + failure reason via the task YAML), and fans out to every adapter whose `Supports(kind)` returns true and whose per-project mute does not include the source `ProjectID` — adapters expose mute via the `IsProjectMuted(projectID) bool` interface, checked before Send so muted projects never reach the network. Per-adapter retry policy: `DefaultRetryDelays = [500ms, 2s, 8s]` (3 attempts total — initial Send + 2 retries — keeping the per-event upper bound at ~10s before the breaker takes over); each failure logs WARN with the adapter id + attempt number and a final `relay_send_failed` ERROR line on retry exhaustion. Per-adapter circuit breaker: a rolling 5-minute window (`CircuitBreakerWindow`) tolerates `CircuitBreakerThreshold = 3` hard failures before opening; subsequent Sends inside the window short-circuit with a single WARN ("breaker open, skipping send") rather than flooding the daemon log when a misconfigured Slack URL or webhook secret means every send dies the same way. The breaker state is keyed by adapter ID, so one flaky endpoint does not block the others. The factory pattern (`AdapterFactory func() ([]Adapter, error)`) is rebuilt by the server's `processWatcherEvents` on `EventIntegrationsChanged`, so URL / mute / event-bitmask edits to `~/.watchfire/integrations.yaml` take effect on the next emit without a daemon restart — the breaker state for adapters that disappeared from the config is cleared so a future re-add starts with a clean slate. The new generic `WebhookAdapter` in `internal/daemon/relay/webhook.go` is the simplest concrete adapter exercising every code path end-to-end: marshal Payload to JSON, sign with the endpoint's secret via `relay.SignHMAC(secret, body)` (returning the canonical `sha256=<hex>` value the receiver verifies via `relay.VerifyHMAC`), POST with `Content-Type: application/json` + `X-Watchfire-Signature: sha256=<hex>` + `X-Watchfire-Event: <KIND>` + `User-Agent: watchfire/<version>` headers, treat 2xx as success and 4xx/5xx as failure. A missing keyring entry on a SecretRef-bearing endpoint fails loudly (`secret not resolved (keyring miss?)`) rather than emitting an unsigned send, so receivers configured to require a signature don't silently drop events. The HMAC signer (`internal/daemon/relay/sign.go`) ships `SignHMAC(secret, body []byte) string`, `VerifyHMAC(secret, body []byte, candidate string) bool` (constant-time compare via `hmac.Equal` — recovering the secret from a probe-loop timing side channel is not viable), and the `SignaturePrefix = "sha256="` constant so future algorithm bumps can land as additional accepted prefixes. The bus's `Subscribe(...) (<-chan Notification, func())` API is unchanged from v5.0 — adapters attach via the existing fan-out, and the dispatcher uses `subscribe-before-Run` ordering so emits during construction land in the 16-slot buffered channel rather than getting dropped. The notification payload schema is pinned at `Payload.Version = 1` and documented in `internal/daemon/relay/payload.go`: `version` / `kind` / `emitted_at` / `project_id` / `project_name` / `project_color` / `task_number` / `task_title` / `task_failure_reason` (TASK_FAILED only) / `deep_link` / `digest_date` (WEEKLY_DIGEST only) / `digest_path` / `digest_body`. Receivers MUST ignore unknown fields; adding a field is a minor compatibility bump (Version stays 1), removing or repurposing one requires a Version=2 cutover. The integrations storage layer (`internal/models/integrations.go` + `internal/config/integrations.go`) was already in place from the settings UI task (0066); v7.0 task 0062 swaps the file-only secret store for an OS-keyring-first chain via `github.com/zalando/go-keyring`: `internal/config/keyring.go::newKeyringStore` probes the platform keyring (Keychain / Credential Manager / dbus secret-service) with a transient set/get/delete cycle so a missing dbus daemon (Linux without a desktop session) or a locked Keychain (macOS over SSH) surfaces at construction time and `chooseSecretStore` falls back to the existing `~/.watchfire/.secrets/` file store with a single one-shot WARN ("OS keyring unavailable (...) — falling back to file store"). Tests inject in-memory fakes via the existing `SetSecretStoreForTest` mechanism so no test ever touches the real platform keyring. The `IntegrationsService.TestIntegration` RPC routes the WEBHOOK kind through the real `relay.WebhookAdapter` (`internal/daemon/server/integrations_service.go::deliverWebhookTest`) — one POST per notification kind, so a single click of "Test" exercises the HMAC signing path + every canonical Payload shape the dispatcher would emit. The CLI's `watchfire integrations test` now accepts the single-arg form mandated by the spec (`watchfire integrations test <id>`) which auto-detects the kind by scanning the configured integrations across webhook → slack → discord → github; the existing two-arg form (`watchfire integrations test <kind> <id>`) is still accepted for callers that want to pin the kind explicitly. `watchfire integrations list` is unchanged. Tests: `internal/daemon/relay/sign_test.go` covers HMAC golden values for non-empty + empty inputs, determinism, secret + body sensitivity, `VerifyHMAC` round-trip, and seven rejection paths (wrong secret / wrong body / empty sig / missing prefix / bad prefix / bad hex / wrong length); `internal/daemon/relay/webhook_test.go` covers the happy path with server-side signature verification, the unsigned-when-no-SecretRef path, the missing-secret-fails-loudly path (signs-or-fails-but-never-silently-skips), HTTP 4xx and 5xx error paths, context cancellation, retry-with-eventual-success through the dispatcher, retry-exhaustion-records-breaker-failure, and the spec's 4-strikes-out circuit breaker that short-circuits the fourth send inside the window with no further receiver hits; `internal/daemon/relay/dispatcher_test.go` covers fan-out only to adapters whose `Supports(kind)` matches, project-mute short-circuit before Send, Reload swapping the adapter slice live, the breaker rolling-window slide (3 failures inside the window opens the breaker; a 6-minute clock advance closes it again), and stop-on-context-cancel; `internal/daemon/notify/bus_subscribe_test.go` adds focused multi-subscriber-fan-out, sibling-cancel-isolation, concurrent emit + subscribe stress, and subscribe-before-emit guarantees on top of the existing bus tests; `internal/config/integrations_test.go` round-trips webhook secrets + Slack URLs through the keyring fake (asserting the YAML on disk never carries the secret), exercises the missing-file-returns-empty path, the keyring-miss-still-loads path (the SecretRef stays so the dispatcher detects the miss and refuses to send rather than silently dropping events), and the delete-clears-keyring path. The dispatcher is wired into the daemon at `internal/daemon/server/server.go` startup with a `context.WithCancel` so `Server.Stop()` cancels the run context (nudging any in-flight Send out of its retry sleep) before calling `dispatcher.Stop()`. `make build` succeeds; `go test ./internal/daemon/relay/... ./internal/daemon/notify/... ./internal/config/... ./internal/daemon/server/...` passes; the daemon starts cleanly with an empty `integrations.yaml` (no adapters means an empty fan-out, no errors on Reload). Out of scope for task 0062: the Slack and Discord adapters (separately shipped as tasks 0063 / 0064 — they predate this task's framework but already conform to the `Adapter` interface), the GitHub auto-PR creator (task 0065), the GUI / TUI settings UI (task 0066), and inbound webhooks (PR merge → close task) which are explicitly deferred from v7.0.

### Fixed

- **GUI: switching projects silently killed every running shell in the bottom panel** — opening project A, spawning a couple of long-lived shells (e.g. `tail -f`, `npm run dev`), then clicking project B in the sidebar destroyed every PTY session — returning to A showed an empty terminal panel with no scrollback and no live processes. Root cause was a documented but unwanted behaviour, not a regression: `gui/src/renderer/src/views/ProjectView/ProjectView.tsx:104-109` ran `useTerminalStore.getState().destroyAllSessions()` from a cleanup effect on `projectId` change, and the Cmd+` toggle (`ProjectView.tsx:88-102`) treated "panel currently shows tabs" as a signal to call the same `destroyAllSessions()` — so collapsing the panel also nuked the shells. Architecturally this flips the bottom panel from "scoped to the active project view" to "global session pool with per-project filtering," which is the only model where long-lived dev servers and tail loops can coexist with project hopping. `ARCHITECTURE.md:907` is updated to match the new contract

### Changed

- **Dashboard auto-sorts projects by activity** — the project grid (and dense list layout) on the Dashboard no longer renders projects in raw `~/.watchfire/projects.yaml` `position` order. The predicate helpers (`hasFailedTask`, `hasReadyTask`, `isProjectWorking`, `isProjectIdle`, `projectActivityGroup`, `sortProjectsByActivity`, `projectOrderDiffers`) live in `dashboard-filters.ts` precisely so task 0036's aggregate status bar and task 0037's filter chips compute their counts from the same definitions — drift between "needs attention" on a card border, in the status bar, in a chip count, and in the auto-sort grouping is now structurally impossible

### Fixed

- **`watchfire update` failed across filesystems on Linux (#25)** — a Fedora user reported `failed to update CLI: install new binary: rename /tmp/watchfire-update-2058306240 /home/madoke/.local/bin/watchfire: invalid cross-device link` when upgrading from 0.9.0 to 1.0.0. Root cause: `updater.DownloadAsset` staged the downloaded binary under `os.TempDir()` (on Fedora/Ubuntu `/tmp` is a separate `tmpfs` filesystem) and `updater.ReplaceBinary` finished the install with `os.Rename(newPath, destPath)`, which on Linux boils down to the `renameat2(2)` syscall and returns `EXDEV` across filesystems. Fix: `DownloadAsset` now takes a `preferDir` argument and stages the download inside the install directory itself (e.g. `~/.local/bin/.watchfire-update-XXXXXX`) so the final rename is always same-filesystem and therefore atomic; the CLI `update` command resolves the install dirs for both the CLI (`os.Executable()` + `EvalSymlinks`) and the daemon (`findDaemonBinary` + `EvalSymlinks`) up front and passes each one as the preferred staging dir. As belt-and-suspenders, `ReplaceBinary` keeps its atomic-rename semantics even if a future caller ever stages the binary elsewhere: it detects the cross-dir case, copies+fsyncs the staged binary into the install dir, removes the original, and finishes with a same-dir rename — the final swap is still a single atomic operation so a concurrent `watchfire` invocation never observes a half-written binary. Exec perms (`0o755`) are applied before the rename so the final binary lands already executable. Regression test in `internal/updater/updater_test.go` covers the same-dir fast path, the cross-dir EXDEV fallback (no leftover staging file, original temp cleaned up), exec-perm preservation, and the `preferDir` fallback to `os.TempDir()` when the install dir is unusable. The daemon binary update path goes through the same `DownloadAsset`/`ReplaceBinary` pair so it inherits the fix automatically. Fixes #25
- **Task list rotated in projects with many tasks (#28)** — a Fedora user with 47 tasks reported the task list starting at `0017`, running to `0047`, then wrapping back to `0001`. Root cause: the task manager's `ListTasks` sorted primarily by the legacy `position` field and only used `task_number` as a tiebreaker, and the TUI re-sorted within each status group with the same key. When a project had tasks split across status groups (16 done + 31 ready in the reporter's case), the section headers — `Ready` renders before `Done` in the TUI, and `In Development` before `Done` in the GUI — placed `0017…0047` above `0001…0016`, producing the "rotation" despite position and task_number matching. Sorting is now canonical across every surface: the task manager returns tasks strictly descending by `task_number` (newest first), the CLI `task list`, TUI task list, and GUI TasksTab all rely on that order without re-sorting, and the legacy `position` field is ignored at read time. Regression test in `internal/daemon/task/manager_test.go` seeds 25 mixed-status tasks with shuffled `position` values and asserts the returned order is strictly descending by `task_number`. Fixes #28
- **GUI prompted to update CLI on every launch (#30)** — the `cli-installer` version check compared the installed CLI's version string against `app.getVersion()` with strict `!==`, which tripped on trailing whitespace, pre-release suffixes, and ANSI hyperlinks that escaped the old CSI-only stripper; on Linux it also read the wrong binary because the search-dir order (`/usr/local/bin` → `~/.local/bin`) put the system path ahead of the user path that `installCLI()` actually writes to, so a stale 2.0.0 binary in `/usr/local/bin` kept re-triggering the prompt even after the user clicked Install. Version parsing now lives in a pure `gui/src/main/version.ts` module with a broader ANSI stripper (CSI + OSC + other ESC), a proper semver-aware comparator (trims whitespace, drops leading `v`, ignores build metadata), and the Linux/macOS search order now matches the install target with a PATH fallback (`command -v`) for rpm/deb/Linuxbrew installs. Regression tests in `gui/src/main/version.test.mjs` cover the representative Linux and macOS outputs (run with `node --test`)
- **Newly-installed agents invisible in GUI/TUI pickers (#29)** — a Fedora user installed Codex while Watchfire was running and neither the TUI nor the GUI surfaced it in the agent picker until `project.yaml` was hand-edited. The architectural root cause was that agent discovery had been implicitly coupled to binary availability — a backend whose `ResolveExecutable` returned an error could get hidden entirely, so a freshly-installed CLI stayed invisible until the resolver happened to find it and the UI re-enumerated. The registry is now the sole source of truth for pickers: `SettingsService.ListAgents` always returns every registered backend (Claude Code, Codex, Gemini, opencode, Copilot) and exposes a new `AgentInfo.available` boolean as a display-time hint only. GUI (`SettingsTab`, `AgentPathsSection`, `DefaultsSection`, `StepAgent`) and TUI (`settings.go`, `taskform.go`, `globalsettings.go`) pickers now append a `(not installed)` suffix to unavailable agents instead of omitting them, so users can select a backend they're mid-installing and get a clear error at spawn time rather than a silent absence. Linux fallback paths for Claude, Codex, Gemini, opencode, and Copilot were simultaneously broadened to cover `/usr/bin/<name>` (distro packages like Fedora `dnf`) and `~/.npm-global/bin/<name>` where they were missing. Regression test in `internal/daemon/server/settings_service_test.go` registers a fake backend whose binary always fails to resolve and asserts it still appears in `ListAgents` with `Available=false`. Fixes #29

## [0.1.0] Ember

Watchfire orchestrates coding agent sessions (starting with Claude Code) based on project specs and tasks. Define what you want built, break it into tasks (or have agents do it), and let agents work through them autonomously — with full visibility into what's happening. Or just turn on wildfire mdoe and let you agents do it all for you.

### Daemon (`watchfired`)

The always-on backend that manages everything:

- **Agent orchestration** — Spawns coding agents in sandboxed PTYs with terminal emulation, one task per project, multiple projects in parallel
- **Git worktree isolation** — Each task runs in its own worktree (`watchfire/<task_number>`), auto-merged back on completion with conflict detection
- **macOS sandbox** — Agents run inside `sandbox-exec` with restricted filesystem/network access
- **File watching** — Real-time detection of task completion and phase signals via fsnotify, with polling fallback for reliability
- **Session logs** — Agent sessions logged to `~/.watchfire/logs/` with JSONL transcripts from Claude Code (clean conversation format) and PTY scrollback fallback
- **System tray** — Menu bar icon showing daemon status, active agents with colored project dots, and quick stop/quit actions
- **Secrets folder** — `.watchfire/secrets/instructions.md` for providing agents with external service credentials and setup instructions, injected into the system prompt
- **Issue detection** — Monitors agent output for auth errors (401, expired tokens) and rate limits (429), with real-time notifications to clients
- **gRPC + gRPC-Web** — Single port serves both native gRPC (CLI/TUI) and gRPC-Web (Electron GUI)
- **Auto-discovery** — Writes connection info to `~/.watchfire/daemon.yaml` so clients find it automatically

### CLI (`watchfire`)

Project-scoped command-line interface:

- `watchfire init` — Initialize a project (git setup, `.watchfire/` structure, `.gitignore`, interactive config)
- `watchfire task add|list|edit|delete|restore` — Full task CRUD with soft delete/restore
- `watchfire definition` — Edit project definition in `$EDITOR`
- `watchfire settings` — Configure project settings interactively
- `watchfire agent start [task|all]` — Start agent in chat, single-task, or run-all-ready mode
- `watchfire agent wildfire` — Autonomous three-phase loop: execute ready tasks → refine drafts → generate new tasks → repeat
- `watchfire agent generate definition|tasks` — One-shot generation commands
- `watchfire daemon start|status|stop` — Daemon lifecycle management
- `watchfire update` — Self-update from GitHub Releases
- **Terminal attach** — Raw PTY streaming with resize handling and Ctrl+C forwarding
- **Self-healing project index** — Auto-registers projects, updates moved paths, reactivates archived projects

### TUI (`watchfire` with no args)

Interactive split-view terminal interface:

- **Split layout** — Task list (left) + agent terminal (right) with draggable divider
- **Left panel tabs** — Tasks (grouped by status), Definition (read-only + `$EDITOR`), Settings (inline form)
- **Right panel tabs** — Chat (live agent terminal), Logs (session history viewer)
- **Agent modes** — Chat, task, start-all, and wildfire with phase display (Execute/Refine/Generate)
- **Issue banners** — Auth required and rate limit detection with recovery guidance
- **Keyboard navigation** — Vim-style (`j/k`), arrows, tab switching (`1/2/3`), panel focus (`Tab`)
- **Mouse support** — Click to focus/select, scroll, drag divider to resize
- **Task management** — Add, edit, status transitions (draft/ready/done), soft delete — all from the keyboard
- **Auto-reconnect** — Reconnects to daemon on disconnect with status indicator
- **Help overlay** — `Ctrl+h` for full keybinding reference

### GUI (Electron)

Multi-project desktop application:

- **Dashboard** — Project cards with task counts, status dots, active task display
- **Project view** — Tasks, Definition, Secrets, Trash, Settings tabs with collapsible right panel (Chat, Branches, Logs)

### Agent Modes

| Mode | Description |
|------|-------------|
| **Chat** | Free-form conversation with the agent at project root |
| **Task** | Work on a specific task in an isolated worktree |
| **Start All** | Run all ready tasks in sequence, one at a time |
| **Wildfire** | Fully autonomous loop: execute → refine → generate → repeat until done |
| **Generate Definition** | One-shot: agent analyzes codebase and writes project definition |
| **Generate Tasks** | One-shot: agent reads definition and creates task files |

### Task Lifecycle

```
draft → ready → done (success ✓ or failure ✗)
```

- Tasks are YAML files in `.watchfire/tasks/`
- Agents detect completion by writing `status: done` to the task file
- Daemon auto-merges the worktree branch, cleans up, and chains to the next task
- Merge conflicts abort the chain to prevent cascading failures

## Generate Tasks Mode

You are generating tasks for a Watchfire project.
You are running at the project root. Do NOT make any code changes.

### Instructions

1. Read the project definition in `.watchfire/project.yaml`.
2. Read all existing tasks in `.watchfire/tasks/` (if any).
3. Analyze the codebase to understand current state and what needs to be done.
4. Create new task YAML files in `.watchfire/tasks/`:
   - Use the next available task number (check existing files and project.yaml next_task_number)
   - Each file: `<4-digit-number>.yaml` (e.g., `0001.yaml`)
   - Required fields ONLY: `version: 1`, `task_id` (8-char alphanumeric), `task_number`,
     `title`, `prompt`, `acceptance_criteria`, `status: ready`, `position` (integer),
     `agent_sessions: 0`
   - **Do NOT include any timestamp fields** — no `created_at`, `started_at`,
     `completed_at`, `updated_at`, or `deleted_at`. The daemon fills timestamps
     in automatically when the task runs; an empty-string timestamp like
     `started_at: ""` is rejected by the YAML parser and would silently halt
     the chain.
   - **ALWAYS single-quote the `title:` value** and double any literal single
     quotes inside it. Titles often contain colons, em-dashes, double quotes, or
     apostrophes, and an unquoted scalar with a literal `: ` (colon-space) is
     parsed as a nested mapping and rejected with `mapping values are not
     allowed in this context`. The bad task is then silently invisible to the
     chain. Correct form:
         title: 'Write a blog post — "Headline: Subhead"'
         title: 'Add /pricing — costs, plans, FAQ'
         title: 'Operator''s guide: reading the daemon logs'
     Wrong form (will be silently dropped):
         title: Write a blog post — "Headline: Subhead"
   - For multi-line scalars in `prompt:` and `acceptance_criteria:`, use a
     block scalar (`|` then indented body) — that form is colon-safe.
5. **After creating tasks, update `next_task_number` in `.watchfire/project.yaml`** to be one more than the highest task number you created. Do NOT modify any other field in `project.yaml`.
6. If no meaningful work remains, create no tasks.
7. **IMPORTANT**: If you modify `.gitignore`, only ADD entries — never remove `.watchfire/` or any existing lines.
8. **Signal completion**: Create an empty file at `.watchfire/tasks_done.yaml`
